# Проект «Каталог обзоров техники» - Бизнес-требования и правила

## 1. Цель проекта
Создать веб-сайт, на котором публикуются экспертные обзоры различных моделей техники (смартфоны, наушники, ноутбуки и т.п.). Посетители могут просматривать каталог устройств, читать подробные рецензии, узнавать плюсы и минусы каждой модели, а также знакомиться с её техническими характеристиками. Проект ориентирован на формирование портфолио и должен демонстрировать навыки проектирования, работы с базой данных, бэкендом и фронтендом.

---

## 2. Целевая аудитория и пользовательские сценарии
**Роли пользователей:**
- **Гость (неавторизованный посетитель):** просматривает список устройств, фильтрует/ищет, открывает страницу устройства с обзором.
- **Администратор (или автор контента):** добавляет новые устройства, редактирует/удаляет их, пишет и публикует обзоры, управляет плюсами/минусами и характеристиками.

**Основные сценарии:**
1. Посетитель заходит на главную - видит список последних обзоров - выбирает интересующее устройство - читает подробный обзор, плюсы/минусы, характеристики.
2. Посетитель ищет конкретную модель через поиск или фильтры по категориям/брендам.
3. Администратор входит в админ-панель - добавляет новое устройство (заполняет бренд, модель, категорию, изображение и пр.) - затем создаёт для него обзор с текстом и списком плюсов/минусов.

---

## 3. Бизнес-правила

### 3.1. Общие правила
- Каждое устройство **должно** принадлежать одной из категорий: «Смартфоны», «Наушники», «Ноутбуки», «Планшеты», «Умные часы» (список может расширяться).
- Каждое устройство **должно** иметь уникальную комбинацию «Бренд + Модель» (например, «Honor 7A» не может быть добавлен дважды).
- Устройство считается доступным для просмотра на сайте только после того, как для него создан и опубликован хотя бы один обзор.

### 3.2. Управление устройствами
- Для добавления устройства администратор **обязан** указать:
  - Бренд (например, Honor, Samsung)
  - Модель
  - Категория
  - Основное изображение (опционально, но рекомендуется)
  - Год выпуска (опционально)
- Остальные характеристики (процессор, память, экран и т.д.) могут быть добавлены позже в виде пар «ключ-значение» (динамические характеристики).
- При удалении устройства все связанные с ним обзоры и характеристики также удаляются (каскадное удаление).

### 3.3. Управление обзорами (рецензиями)
- Обзор **обязательно** привязан к одному устройству.
- Обзор должен содержать:
  - Заголовок (например, «Подробный обзор Honor 7A: стоит ли своих денег?»)
  - Текст обзора (основное содержание)
  - Автор (имя или псевдоним, по умолчанию «Эксперт»)
  - Дата публикации (проставляется автоматически)
- Обзор может иметь **список плюсов** и **список минусов** (каждый пункт - отдельная запись, не более 200 символов).
- Плюсы и минусы отображаются отдельными блоками на странице устройства.
- У одного устройства может быть **только один активный (опубликованный) обзор**. Если требуется новый обзор, старый должен быть снят с публикации или помечен как черновик. (Примечание: для упрощения можно разрешить несколько обзоров, но в рамках учебного проекта лучше ограничиться одним, чтобы не усложнять выборку.)

### 3.4. Характеристики устройства
- Технические характеристики хранятся в виде гибкого списка «название: значение» (например, «Экран: 5.45 дюймов», «Процессор: Snapdragon 430»).
- Администратор может добавлять, удалять и редактировать характеристики для каждого устройства.
- На странице устройства характеристики выводятся в виде таблицы или списка.

### 3.5. Визуализация и интерфейс
- На главной странице отображается сетка или список устройств с обложкой, названием и кратким превью обзора.
- При клике на устройство открывается детальная страница, где показаны: полный обзор, блок плюсов/минусов, все характеристики, изображение устройства.
- Должна быть реализована пагинация или бесконечная прокрутка для списка устройств (например, по 10 на странице).
- Поиск осуществляется по названию модели, бренду и тексту обзора (простейший поиск без полнотекстовых индексов, но можно использовать `LIKE`).

## 4. Ограничения и нефункциональные требования
- Система должна быть реализована на стеке технологий: Python/Django.
- Данные хранятся в реляционной базе данных (SQLite, PostgreSQL, MySQL (Тут я еще не решил, но обычно берём PostgreSQL, но в этот раз у меня есть мысль поменять)).
- Интерфейс администратора может быть либо отдельной страницей с формами, либо использовать готовую админ-панель (например, Django Admin).
- Код должен быть структурирован, содержать комментарии и соответствовать принципам ООП (где уместно).

## 5. Пример описания бизнес-процесса «Добавление нового обзора»
1. Администратор авторизуется в системе.
2. Переходит в раздел «Устройства» и нажимает «Добавить устройство».
3. Заполняет поля: бренд, модель, категория, загружает изображение.
4. После сохранения устройства система перенаправляет на форму создания обзора.
5. Администратор вводит заголовок, текст обзора, список плюсов и минусов (каждый с новой строки или через специальное поле).
6. Нажимает «Опубликовать». Система проверяет, нет ли уже опубликованного обзора для этого устройства. Если есть - предупреждает, что предыдущий обзор будет снят с публикации (или архивирован).
7. После подтверждения обзор сохраняется, становится активным и появляется на сайте.
