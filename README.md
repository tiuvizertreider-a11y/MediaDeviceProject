# Проект «Каталог обзоров техники» - Бизнес-требования (редакция 2)

## 1. Цель проекта
Создать веб-сайт, на котором публикуются экспертные обзоры различных моделей техники (смартфоны, наушники, ноутбуки и т.п.). Посетители могут просматривать каталог устройств, читать подробные рецензии, узнавать плюсы и минусы каждой модели, а также знакомиться с её техническими характеристиками. Проект ориентирован на формирование портфолио и должен демонстрировать навыки проектирования, работы с базой данных, бэкендом и фронтендом.

---

## 2. Целевая аудитория и пользовательские сценарии

**Роли пользователей:**
- **Гость (неавторизованный посетитель):** просматривает список устройств, фильтрует/ищет, открывает страницу устройства с обзором.
- **Суперпользователь (администратор):** управляет всеми данными через встроенную админ-панель Django: добавляет, редактирует и удаляет устройства, обзоры, характеристики, плюсы/минусы. Других ролей на данном этапе не предусмотрено.

**Основные сценарии:**
1. Посетитель заходит на главную - видит список последних обзоров (устройств с опубликованными обзорами) - выбирает интересующее устройство - читает подробный обзор, плюсы/минусы, характеристики.
2. Посетитель ищет конкретную модель через поиск (по названию, бренду, тексту обзора) или фильтрует по категориям.
3. Администратор входит в админ-панель - добавляет новое устройство (заполняет бренд, модель, категорию, изображение и пр.) - затем создаёт для него обзор с текстом и списком плюсов/минусов, при необходимости публикует его (при этом предыдущий опубликованный обзор этого устройства автоматически архивируется).

## 3. Бизнес-правила

### 3.1. Общие правила
- Каждое устройство **должно** принадлежать одной из категорий: «Смартфоны», «Наушники», «Ноутбуки», «Планшеты», «Умные часы» (список может расширяться).
- Каждое устройство **должно** иметь уникальную комбинацию «Бренд + Модель» (например, «Honor 7A» не может быть добавлен дважды).
- Устройство считается доступным для просмотра на сайте только после того, как для него создан и опубликован **хотя бы один обзор** со статусом «опубликован».

### 3.2. Управление устройствами
- Для добавления устройства администратор **обязан** указать:
  - Бренд (например, Honor, Samsung)
  - Модель
  - Категория
  - Основное изображение (опционально, но рекомендуется)
  - Год выпуска (опционально)
- Остальные характеристики (процессор, память, экран и т.д.) могут быть добавлены позже в виде пар «ключ-значение» (динамические характеристики). Администратор может добавлять, удалять и редактировать характеристики для каждого устройства.
- При удалении устройства все связанные с ним обзоры и характеристики также удаляются (каскадное удаление).

### 3.3. Управление обзорами (рецензиями)
- Обзор **обязательно** привязан к одному устройству.
- Обзор должен содержать:
  - Заголовок (например, «Подробный обзор Honor 7A: стоит ли своих денег?»)
  - Текст обзора (основное содержание)
  - Автор (имя или псевдоним, по умолчанию «Эксперт»)
  - Дата публикации (проставляется автоматически)
  - Статус: черновик, опубликован, архив.
- Обзор может иметь **список плюсов** и **список минусов** (каждый пункт - отдельная запись в базе данных, не более 200 символов). Плюсы и минусы отображаются отдельными блоками на странице устройства в порядке добавления.
- **У одного устройства может быть только один активный (опубликованный) обзор.** Если администратор публикует новый обзор, предыдущий опубликованный автоматически переводится в статус «архив». Архивные обзоры хранятся в базе данных, но не показываются на сайте. Черновики также не отображаются.

### 3.4. Характеристики устройства
- Технические характеристики хранятся в виде гибкого списка «название: значение» (например, «Экран: 5.45 дюймов», «Процессор: Snapdragon 430»).
- Администратор может добавлять, удалять и редактировать характеристики для каждого устройства через inline-формы в админ-панели.
- На странице устройства характеристики выводятся в виде таблицы или списка (порядок - по названию характеристики).

### 3.5. Визуализация и интерфейс
- **Главная страница:** отображается сетка или список устройств, для которых есть опубликованный обзор. У каждого устройства показывается обложка, название (бренд + модель), возможно краткое превью обзора.
- **Детальная страница устройства:** полный обзор (только опубликованный), блок плюсов/минусов, все характеристики, изображение устройства.
- **Пагинация:** список устройств разбит на страницы (например, по 10 на странице).
- **Поиск:** поле поиска, которое ищет по названию модели, бренду, заголовку обзора и тексту обзора. Используется простое частичное совпадение (`icontains`).
- **Фильтрация:** выпадающий список категорий (и, возможно, другие фильтры, например, по брендам). Фильтрация работает в сочетании с поиском.
- **Интерфейс администратора:** используется стандартная админ-панель Django с кастомизацией (inline-формы для характеристик и плюсов/минусов, действия для публикации с архивацией предыдущих обзоров). Отдельные страницы для администратора не разрабатываются.

### 3.6. Управление изображениями
- Изображения устройств загружаются через админ-панель и сохраняются локально в папку `media/`. Для работы с изображениями используется библиотека Pillow.
- В production (при использовании Docker) папка с медиафайлами монтируется как том, чтобы данные не терялись при пересборке контейнера.

## 4. Ограничения и нефункциональные требования

### 4.1. Технологический стек
- **Язык программирования:** Python 3.14+
- **Фреймворк:** Django
- **База данных:** PostgreSQL
- **Контейнеризация:** Docker, docker-compose
- **Тестирование:** pytest, цель покрытия не менее 70%
- **Стиль кода:** PEP 8, используется pylint для статического анализа.

### 4.2. Качество кода и стандарты разработки
- Код должен быть структурирован, содержать комментарии и соответствовать принципам ООП.
- Обязательно наличие unit-тестов для моделей (валидация данных, бизнес-логика) и интеграционных тестов для представлений (доступность страниц, поиск, фильтрация).
- Все зависимости фиксируются в `requirements.txt`.

### 4.3. Инфраструктура и развертывание
- Проект должен быть полностью подготовлен к запуску в изолированной среде с использованием Docker.
- Наличие `Dockerfile` для сборки образа приложения.
- Наличие `docker-compose.yaml` для оркестрации сервисов: приложение (Django) и база данных (PostgreSQL).
- Переменные окружения (секретный ключ, настройки БД) должны храниться в файле `.env` (не включается в репозиторий).
- Простота запуска: одна команда `docker-compose up --build` должна поднимать всё окружение, выполнять миграции и делать приложение доступным на `http://localhost:8000`.

## 5. Пример описания бизнес-процесса «Добавление нового обзора»

1. Администратор авторизуется в админ-панели Django (`/admin`).
2. Переходит в раздел «Устройства» и нажимает «Добавить устройство».
3. Заполняет поля: бренд, модель, категория, загружает изображение (опционально), при необходимости добавляет характеристики через inline-форму.
4. После сохранения устройства переходит в раздел «Обзоры» и создаёт новый обзор, выбирая устройство из списка.
5. Вводит заголовок, текст обзора, список плюсов и минусов (каждый пункт добавляется через inline-форму).
6. Устанавливает статус «Черновик» или «Опубликован». Если выбирает «Опубликован», система автоматически находит все опубликованные обзоры для этого же устройства и переводит их в статус «Архив».
7. После сохранения обзор становится видимым на сайте (если опубликован).
